### ❌ Divergências / gaps (em ordem de impacto)

**🔴 Grande (features inteiras que não existem):**

1. **Context references** — `@file:path`, `@folder:path`, `@diff`, `@staged`, `@git:N`, `@url:URL` (com line ranges `@file:foo.py:10-25`). Hermes tem, a gente não tem nada. É o feature mais visível que o usuário final sentiria falta — qualquer pessoa acostumada com Claude Code manda "revisa o `@diff`" ou "olha o `@file:main.py`" naturalmente.

2. **Memory tool pro agente** — `add` / `replace` / `remove` com substring matching. Hermes expõe como tool pro LLM chamar. Hoje a gente só tem o write manual via wizard/UI, sem o self-improvement loop entre sessões.

3. **Progressive subdirectory discovery** — Hermes injeta AGENTS.md/CLAUDE.md/.cursorrules de subdirs na hora que o agente navega pra lá. A gente carrega só o do CWD.

**🟡 Médio (comportamento Hermes que a gente ignora):**

4. **`write_approval` gate** — `memory.write_approval: true|false` no config. Quando `true`, todo save de memory (incluindo do background self-improvement review) fica staged pra approval. Importante porque sem o gate, o review pode escrever sem pedir.

5. **Memory full error** — quando o agent tenta escrever além do char limit, Hermes retorna erro com a lista de `current_entries` e o agent resolve consolidando. A gente provavelmente trunca silencioso.

6. **Duplicate prevention** — Hermes auto-rejeita entries duplicadas exatas. A gente não.

7. **Soft/hard limit pros context references** (25% / 50% do context length) — Hermes avisa e depois recusa expansão. Sem isso, `@file:big.json` pode estourar o contexto.

8. **Sensitive path blocking** (`~/.ssh`, `~/.aws`, `~/.gnupg`, `~/.kube`, `$HERMES_HOME/skills/.hub/`) — crítico pra segurança do `@file:`.

9. **Binary file detection** (MIME type + null-byte scan) — Hermes rejeita binário com warning.

10. **Path traversal protection** — paths fora do workspace root são rejeitados. (Já temos workspace bound, mas o check explícito ainda não existe no contexto de `@file:`).

**🟢 Cosmético / nice-to-have:**

11. **Mais personality presets** — Hermes tem 14 (helpful/concise/technical/creative/teacher/kawaii/catgirl/pirate/shakespeare/surfer/noir/uwu/philosopher/hype). A gente tem 5.
12. **`agent.personalities` no `config.yaml`** — custom personalities (e.g. `codereviewer`) definidas pelo usuário.
13. **CLAUDE.md / .cursorrules / .cursor/rules/*.mdc** detection — Hermes aceita esses formatos; a gente só lê os 4 próprios (`SOUL/IDENTITY/USER/MEMORY`).
14. **External memory providers** (Honcho, OpenViking, Mem0, Hindsight, Holographic, RetainDB, ByteRover, Supermemory) — fora do escopo do projeto solo.
15. **CLI tab completion pra `@`** — N/A (Tauri UI, não CLI).
16. **`background_review` em model mais barato** (`auxiliary.background_review`) — N/A sem memory tool primeiro.
17. **Memory notifications** (`display.memory_notifications: on/off/verbose`) — depende do #2 (memory tool com background review).

## Plano de execução

### PR A — Context References (PRIORIDADE: alta, BLOQUEIO: nada)

**Escopo:** implementar `@file:`, `@folder:`, `@diff`, `@staged`, `@git:N`, `@url:` no composer (chat + coding) com size limits + security guards. É o feature mais visível pro usuário final.

**Backend (`web/backend/main.py` + novo `web/backend/context_refs.py`):**
- `POST /api/context-refs/expand` recebe `{refs: ["@file:src/main.py", "@file:foo.py:10-25", "@diff", ...], workspace_dir}` → retorna `{ref, content, truncated, warning}` pra cada um
- Path resolution relativo a `workspace_dir`, com reject pra fora do root
- Sensitive path blocklist (`~/.ssh`, `~/.aws`, `~/.gnupg`, `~/.kube`, `.env`, etc.) — retorna `{ref, error: "sensitive path"}`
- Binary detection (null-byte scan + MIME via `mimetypes`) — retorna `{ref, error: "binary"}`
- Soft limit 25% do `model_limit` (warning appended, expansion proceeds) / hard limit 50% (refuse all, original returned)
- Line range support em `@file:` (1-indexed, inclusive)
- Git operations via `subprocess.run(["git", "diff"])` no workspace_dir
- URL fetch via `httpx.get()` (max 50KB, html2text ou basic strip)

**Frontend (`desktop/src/components/composer/`):**
- Novo `<ContextRefAutocomplete>` no composer: ao digitar `@`, abre popover com 6 tipos (`@diff`, `@staged`, `@file:`, `@folder:`, `@git:`, `@url:`) + lista de files/folders do cwd
- Path completion via filesystem walk (limited to 200 entries)
- Em `sendMessage()`, expande refs localmente (chama `/api/context-refs/expand`) ANTES de mandar pro WS
- Mostra preview chip de cada ref expandido: `@file:src/main.py → 247 lines (8.2KB) [attached]`
- Erros viram warning chip: `@file:~/.ssh/id_rsa → ⚠ sensitive path`
- Cap visual: quando soma expandida > 25% do context, banner amarelo no composer; quando > 50%, refuse o send (button disabled)

**Tests:**
- pytest: `tests/test_context_refs.py` — path resolution, sensitive blocklist, binary detection, size limits, line ranges, git ops, URL fetch
- vitest: `ContextRefAutocomplete.test.jsx` — popover open, completion, expansion, error chips, hard limit disable

**Estimativa:** ~600-800 LOC, 1 sprint.

### PR B — Memory Tool pro Agente (PRIORIDADE: alta, BLOQUEIO: nada)

**Escopo:** expor o `memory` tool (add/replace/remove) pro LLM chamar durante a sessão. Destrava o self-improvement loop que o Hermes faz (correções e padrões viram entries persistentes).

**Backend (`mini_agent/tools/memory_tool.py` — novo):**
- Tool com 3 actions: `add` / `replace` / `remove`
- Target: `memory` (MEMORY.md, 2.200 chars) ou `user` (USER.md, 1.375 chars)
- `add` params: `{target: "memory"|"user", content: str}` → append, rejeita duplicate exato, rejeita overflow
- `replace` params: `{target, old_text: str, content: str}` → usa substring matching (Hermes pattern), `old_text` precisa ser substring único de exatamente 1 entry, senão erro
- `remove` params: `{target, old_text: str}` → mesma substring matching
- Capacity check ANTES de qualquer write: se `(current + new) > limit` → erro com lista de `current_entries` e `usage: "X/limit"` (Hermes pattern exato)
- Security scan em `content` (mesma função do `scan_for_injection()` em `agent_context.py`)
- Auditoria: `logger.info("memory_write", extra={target, action, old_chars, new_chars, session_id})`

**Backend (registro no agent):**
- `mini_agent/agent.py`: nova tool `MemoryTool` no set default (junto com `ReadTool`/`WriteTool`/`BashTool`). Skip se `config.minimax.memory_enabled: false`.
- WS `memory_write` event (broadcast pro frontend, opcional) → notifica que a memory foi atualizada
- Frontend recebe → invalida o `useAgentContext().status` (refetch char_usage)

**Frontend (opcional mas nice):**
- `desktop/src/components/agent-context/MemoryAuditLog.jsx` (sessão dedicada) — mostra últimas 20 writes (timestamp, target, action, char delta). Read-only.
- Settings toggle `memory.write_approval: true|false` no Agent Card (mesmo padrão do `auto_compact` toggle). Quando `true`, todo write vira prompt de approval inline (ou staged em `/memory pending` se não-CLI).

**Tests:**
- pytest: `tests/test_memory_tool.py` — add/replace/remove com substring matching, duplicate prevention, overflow error, security scan
- pytest: integration test — agent invoca memory tool durante turn, arquivo persiste, próxima session carrega

**Estimativa:** ~400-500 LOC, 1 sprint.
